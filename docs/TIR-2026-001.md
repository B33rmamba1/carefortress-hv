# CareFortress Threat Intelligence Report — TIR-2026-001

**Classification:** Public — Threat Intelligence Sharing  
**Date:** 2026-07-19  
**Author:** James Smith  
**Status:** Closed — Attacker engaged honeypot infrastructure only, no real systems compromised  

---

## Executive Summary

Between July 16–19, 2026, the CareFortress honeypot infrastructure on carefortress.dev captured two independent automated exploitation pipelines executing credential theft and post-exploitation against fake infrastructure endpoints. Both attackers scraped fabricated credentials from honeypot content, extracted embedded hostnames, probed the fake infrastructure for database admin panels, and in one case tested stolen AWS API keys against real AWS endpoints within 60 seconds of initial harvest.

The most aggressive attacker (`146.70.187.210`) executed the full kill chain — from initial reconnaissance through AWS credential validation — in under one minute, demonstrating a fully automated scrape-to-exploit pipeline running on a Windows Server 2022 VPS with Python/Boto3.

No real systems were compromised. All credentials, hostnames, and infrastructure endpoints served to the attackers were fabricated canary tokens designed to detect and attribute this exact behavior.

---

## Infrastructure Overview

CareFortress runs a multi-layer honeypot on carefortress.dev:

- **Spider Trap (Cloudflare Worker):** Intercepts requests to ~100 known scanner paths (/.env, /.git/config, /wp-login.php, etc). Low-value paths return an infinite maze of random links. High-value paths return realistic fake credentials with embedded per-visitor tracking tokens.
- **Canary Token System:** Each visitor receives a deterministic 8-character hex token ID derived from their IP. Fake credentials embed unique hostnames (`int-{tokenId}.carefortress.dev`, `repo-{tokenId}.carefortress.dev`) that route through a Cloudflare tunnel to a SIEM listener.
- **AWS Canary Token Pool:** 20 real Thinkst canarytokens.org AWS key pairs served in fake `.env` and `.aws/credentials` files. Any AWS API call using these keys triggers a CloudTrail alert with the caller's IP, user agent, and API action.
- **Wazuh SIEM:** All events (spider trap hits, canary callbacks, correlation alerts) feed into Wazuh 4.14.6 with 18 custom detection rules.

| **Sophistication** | Medium-High — fully automated, tests multiple credential profiles independently, uses low-privilege API calls for stealth validation |

**AWS Credential Validation Pattern:**

The attacker's Boto3 tooling follows a specific validation sequence:

1. `sts:GetCallerIdentity` — Verify the key is valid and retrieve the account/ARN
2. `servicequotas:GetServiceQuota` — Probe for service-level permissions
3. `sns:GetSMSAttributes` — Check SNS permissions (common over-provisioned permission)

`GetSMSAttributes` is a known credential validator favorite because:
- It requires `sns:GetSMSAttributes` permission, which many IAM policies accidentally include
- A successful response reveals the account has SMS capability (useful for phishing/2FA bypass)
- It generates minimal CloudTrail noise compared to more obvious enumeration calls

The multi-layer approach captured the complete attack chain:

1. **What they scanned** (spider trap path logs)
2. **What they stole** (CANARY_SERVED events with specific paths and token IDs)
3. **What infrastructure they probed** (canary hostname callbacks with paths and timing)
4. **What credentials they tested** (AWS CloudTrail with API calls, SDK version, OS)
5. **What permissions they sought** (specific AWS API call sequence)

## Recommendations for Defenders

1. **Monitor for `GetSMSAttributes` in CloudTrail.** It's a strong signal for credential stuffing against AWS accounts. Legitimate use is rare.
2. **Alert on `GetCallerIdentity` from unexpected IPs.** It's the universal "is this key alive" check for stolen credentials.
3. **Block the "SecurityResearch/1.0" user agent** if you don't run authorized external scanning. It's used by bulk credential scrapers pretending to be benign.
4. **Treat any .env or .aws/credentials exposure as a credential compromise** — automated pipelines will test stolen keys within 60 seconds.
5. **Deploy canary tokens in honeypot files.** The attribution value (linking harvest IP to exploitation IP, identifying attacker tooling) far exceeds the setup cost.
